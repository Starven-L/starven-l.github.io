Level2 - Fiding Neighbours

先利用service特性ip反解扫一下目前处于的B段有哪些服务

发现reporting-service.k8s-lan-party.svc.cluster.local但是没东西

由于题目描述说Sidecar，前面也专门写了文章学习这个

说明当前容器处于的Pod是还存在其他的容器的（按照题目说就是sidecar容器）

而我们知道The sidecar container shares the same lifecycle, resources, and network namespace as the main container.

hint2提到我们的环境有tcpdump

同时我们知道由于sidecar容器和主容器共享网络命名空间，因此我在主应用程序容器监听流量就可以看到sidecar的流量

可以看到这个通信,这就是在于刚才那个curl没回显内容的一个通信，原来flag藏在流量里面了

拿到flag：wiz_k8s_lan_party{good-crime-comes-with-a-partner-in-a-sidecar}

也就是说，我是在主容器执行的curl reporting-service.k8s-lan-party.svc.cluster.local.http没回显
这个时候在主容器执行tcpdump -X得到刚才这个流量

意思就是sidecar在于reporting-service.k8s-lan-party.svc.cluster.local.http通信

为什么呢？

我所在的 Pod 中使用了 Sidecar 代理（比如 Istio）

  • 我发起的 curl http://reporting-service... 请求，并没有直接发给目标服务；
  • 它被 Pod 中的 Sidecar（例如 Envoy）劫持、代理、重写转发；
  • Sidecar 作为客户端向 reporting-service 发起了真正的请求；

至于用的Envoy代理我们curl也可以看到服务端确实用的istio-envoy

【问题记录★★★★★】：那么我为啥抓流量得到了flag而执行curl http://reporting-service… 请求没有flag

我猜测即使请求是我主容器发出的，但返回的响应（包含 flag） 没有真正返回给主容器，而是返回给了Sidecar容器

但是回到这个题，其实是因为主容器抓包网络流量可以抓到Sidecar的，而题目设置就是Sidecar容器在主动向服务 POST 数据，flag 藏在 POST 请求体中；这里tcpdump只是监听到sidecar容器的流量，也侧面突出证明了sidecar容器和主ap容器共享网络命名空间

题目总结：

这个题通过 Sidecar 容器偷偷 POST 出 flag，主容器无法直接访问 flag，但因为 Pod 内共享网络，可以用 tcpdump 嗅探到 Sidecar 发出的请求，从中看到 flag。

Level3 - Data Leakage

首先还是dns反向扫描一下，看有没有啥服务，但是并没有

题目描述说的跟文件存储有关还有网络共享，S3存储桶这种云存储服务？还有啥网络共享？

仅仅基于网络，也就是说没有身份认证这些功能啥的不需要去找凭据通过creds这些去认证

而题目所指的网路共享其实就是AWS EFS托管的

mount 或者 cat /proc/mounts 查看磁盘挂载

可以看到有一个文件共享挂载到了/efs

fs-0779524599b7d5e7e.efs.us-west-1.amazonaws.com:/ on /efs type nfs4 (ro,relatime,vers=4.1,rsize=1048576,wsize=1048576,namlen=255,hard,noresvport,proto=tcp,timeo=600,retrans=2,sec=sys,clientaddr=192.168.48.9,local_lock=none,addr=192.168.124.98)

但是没有权限

然后了解到有efs相关cli来支持url的一些访问命令操作，有libnfs专门提供nfs的client来操作api

比如说nfs-ls来列举这个网络共享文件系统的文件

有一个权限不足：

  • 默认情况下，NFS 客户端会使用当前用户的 UID 和 GID 来访问文件系统。
  • AWS EFS NFSv4 默认使用 POSIX 权限控制（uid/gid）来决定访问权限。
  • 加上 uid=0&gid=0，以root身份访问，也就绕过了默认用户权限限制。

Level4 - Bypassing Boundaries

还是先dns反向扫描探测一下存活服务，可以看到10.100.224.159有服务的

尝试访问但是权限拒绝

因为权限策略不是只禁止了GET和POST嘛，我就在想如果后端也接受其他方法，那么我就可以通过其他方法来访问

但是不管是OPTION，HEAD，PUT，PATCH都是501，说明后端不支持

最后发现其实考察的是下面这个考点

核心：Istio客户端存在一些bypass方法绕过RBAC

（报告里搜索 Istio Client-Side Bypasses）

  • UDP协议，因为 Istio 不处理 UDP 数据包
  • 1337 用户 UID，Istio的sidecar创建基于UID的iptables规则。如果Pod中的主容器以UID 1337运行就能绕过使用Envoy代理sidecar的iptables规则，规则详情见 https://github.com/istio/istio/wiki/Understanding-IPTables-snapshot#use-pid-to-get-iptables
  • CAP_SETUID 容器权限，当容器具有CAP_SETUID（一个默认权限）时，任何具有该权限的进程都可以将其UID更改为1337，从而实现上述绕过
  • 当容器被授予CAP_NET_ADMIN权限时，它可以重写自己的iptables规则并绕过Envoy代理
  • 入站端口绕过，和这题没什么关系

但是具体分析第二个可以bypass的原因到底是为什么呢？不妨先搞清楚网络通信规则

这个sidecar的部署来由：每个 Pod 中自动注入一个 Envoy Sidecar（通过 istio-injection=enabled 注解）

然后深究其原理其实就是：Istio 在设置 iptables 规则时默认跳过 UID 1337 的进程流量，因此以 UID 1337 运行的进程就能绕过 Envoy，也就绕过了 Istio 的所有安全策略（比如 RBAC、mTLS、审计等）。

回到此题，发现有1337 istio用户

切换用户即可绕过RBAC

Level5 - Lateral Movement

这个策略解读一下呢：其实很通俗易懂，就是提一下mutate作用是自动修改资源内容

一句话解释就是：任何在sensitive-ns命名空间创建的pod都会自动注入环境变量flag

首先又是一个新东西kyverno，这是个啥玩意？

  • Kyverno 是一个为 Kubernetes 设计的 策略引擎（Policy Engine）
  • Kyverno 本质上是一个 Kubernetes Admission Webhook，部署在集群中,工作原理：
    • 用户提交资源（如创建 Pod、Deployment）
    • K8s API Server 调用 Kyverno Webhook
    • Kyverno：
      • 解析策略（Policy）
      • 判断是否符合验证规则、是否需要自动变更、是否要附加资源
    • 拒绝或允许请求，或自动修改资源后放行
